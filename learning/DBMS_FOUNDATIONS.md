## 0. The big picture: where the database sits

```
React SPA (browser)  ──HTTP/JSON──►  FastAPI (Python)  ──SQL──►  PostgreSQL
                                         │
                                    SQLAlchemy ORM
                              (turns Python objects ⇄ SQL rows)
```

The app **never** writes raw SQL by hand for normal operations. We describe tables as
Python classes (the **ORM**, section 11), and SQLAlchemy generates the SQL. PostgreSQL
is the actual database engine that stores the data on disk and enforces the rules.

## 1. The relational model: tables, rows, columns

**Idea.** A relational database stores data in **tables** (also called *relations*). A
table has **columns** (fields, each with a fixed type) and **rows** (records, one per
real-world thing). It's a spreadsheet with strict typing and rules.

**In our code.** Every class in [`models.py`](../backend/app/modules/models.py) that
inherits `Base` is one table. We have 9 today: `users`, `resources`, `recurrence_rules`,
`events`, `event_participants`, `bookings`, `notifications`, `audit_logs`, `feedback`.

```python
class User(Base):
    __tablename__ = "users"          # the table name
    id = Column(...)                 # a column
    email = Column(String(255), ...) # a column with a type + max length
```
One `User(...)` object you create in Python becomes one **row** in the `users` table.

**Learn more:** [Khan Academy – Intro to SQL](https://www.khanacademy.org/computing/computer-programming/sql) · [Postgres: "What is a relation"](https://www.postgresql.org/docs/current/tutorial-concepts.html)

## 2. Schema and data types

**Idea.** The **schema** is the blueprint: which tables exist, which columns, what type
each column is, and what rules apply. Types matter — a date stored as text can't be
compared or sorted reliably.

**In our code.** We use a small set of column types (imported at the top of `models.py`):

| Our type | Means | Example column |
|---|---|---|
| `UUID(as_uuid=False)` | 128-bit unique id stored as text | every `id` |
| `String(255)` | short text, capped length | `email`, `full_name` |
| `Text` | long text, no cap | `description`, `notes` |
| `Boolean` | true/false | `is_active`, `requires_approval` |
| `DateTime(timezone=True)` | timestamp with timezone | `start_time`, `created_at` |
| `Integer` | whole number | `capacity` |
| `Enum` (`SAEnum`) | one value from a fixed list | `role`, `resource_type`, `status` |

**Why timezone-aware datetimes?** A booking at "10:00" is meaningless without a timezone.
We store everything in UTC (`utcnow()` in `models.py`) and the recurrence engine converts
to UTC before comparing — see [`core/recurrence.py`](../backend/app/core/recurrence.py).

**Learn more:** [Postgres data types](https://www.postgresql.org/docs/current/datatype.html)

## 3. Primary keys (PK)

**Idea.** A **primary key** is the column that uniquely identifies each row. No two rows
share it; it's never null. It's how other tables point at this row.

**In our code.** Every table uses a UUID primary key:
```python
id = Column(UUID(as_uuid=False), primary_key=True, default=new_uuid)
```
`new_uuid()` generates a random `uuid4` string. **Why UUIDs instead of 1, 2, 3…?**
- Auto-increment integers leak information (row #5000 tells you the table size).
- UUIDs can be generated by the app *before* hitting the DB, and never collide across
  tables/services — useful if this ever splits into microservices.
- Trade-off: bigger and not naturally ordered (a small cost we accept).

**Learn more:** [Postgres primary keys](https://www.postgresql.org/docs/current/ddl-constraints.html#DDL-CONSTRAINTS-PRIMARY-KEYS) · [UUID vs auto-increment](https://www.cybertec-postgresql.com/en/uuid-serial-or-identity-columns-for-postgresql-auto-generated-primary-keys/)

## 4. Foreign keys (FK) and referential integrity

**Idea.** A **foreign key** is a column that holds the primary key of a row in *another*
table — a pointer that creates a link. **Referential integrity** means the database
refuses to store a pointer to a row that doesn't exist (you can't have a booking for a
deleted resource).

**In our code.** `Event.organizer_id` points at a `users` row:
```python
organizer_id = Column(UUID(...), ForeignKey("users.id"), nullable=False)
```
A `Booking` carries **three** foreign keys at once — it links an event, a resource, and
the person who asked:
```python
event_id     = ForeignKey("events.id")
resource_id  = ForeignKey("resources.id")
requester_id = ForeignKey("users.id")
```
That's the whole "Booking = a claim on a Resource for an Event by a User" idea expressed
as foreign keys.

**Learn more:** [Postgres foreign keys](https://www.postgresql.org/docs/current/ddl-constraints.html#DDL-CONSTRAINTS-FK)

## 5. Relationships: one-to-many and many-to-many

**Idea.** Foreign keys create relationships between tables:

- **One-to-many (1:M)** — one row on the "one" side links to many on the "many" side.
  One `User` organizes many `Event`s. The FK lives on the "many" side (`Event.organizer_id`).
- **Many-to-many (M:N)** — rows on both sides link to many of the other. You can't do this
  with a single FK; you need a **junction table** (a.k.a. join/bridge table) whose rows
  each pair one id from each side.

**In our code.**
- *1:M:* `User → Events`, `User → Bookings`, `Resource → Bookings`, `Event → Bookings`.
  SQLAlchemy exposes these as Python lists via `relationship(...)`:
  ```python
  # on User:
  owned_events = relationship("Event", back_populates="organizer", ...)
  bookings     = relationship("Booking", back_populates="requester", ...)
  ```
  `back_populates` keeps both sides in sync (an Event knows its organizer; a User knows
  its events).
- *M:N:* `event_participants` is a junction table linking `events` ↔ `users` (one row =
  "this user is a participant in this event"), with an RSVP status on the link itself.

> ✅ **Phase 2 update — worked many-to-many example.** These tables now exist in
> `modules/models.py`. Two M:N relationships, each via a **junction table**:
>
> ```
> roster_people ──(group_members)── groups ──(event_groups)── events
> ```
>
> - **person ↔ groups** via `group_members` (rows pair `roster_person_id` + `group_id`).
> - **event ↔ groups** via `event_groups` (rows pair `event_id` + `group_id`).
>
> To list everyone in a group, you "follow" the junction — in SQL that's a **JOIN**
> (your first one):
> ```sql
> SELECT p.full_name
> FROM group_members gm
> JOIN roster_people p ON p.id = gm.roster_person_id
> WHERE gm.group_id = '<group X>';
> ```
> Read it as: walk `group_members`, and for each row **JOIN** to the matching
> `roster_people` row (`p.id = gm.roster_person_id`), keeping only group X.
>
> So "which students does event E touch?" = events → `event_groups` → groups →
> `group_members` → `roster_people`. **Clash on students** = does event A's set of people
> **intersect** event B's set. The `UniqueConstraint`s stop duplicate pairings.

**Learn more:** [Junction tables explained](https://en.wikipedia.org/wiki/Associative_entity) · [SQLAlchemy relationships](https://docs.sqlalchemy.org/en/20/orm/basic_relationships.html)

## 6. Entity-Relationship (ER) modeling

**Idea.** Before tables, you model the domain as **entities** (nouns) and the
**relationships** between them. Good modeling = each entity represents one concept, with
no overlap.

**In our code.** The deliberate three-way split is the project's central modeling decision
(documented at the top of `models.py`):

- **Event** — *what* is happening (a lecture). Owns title, time, organizer.
- **Resource** — a *thing* that can be reserved (room, lab, equipment).
- **Booking** — a *claim* linking an Event to a Resource for a time window.

Keeping them separate is why "one lecture in a room **and** using a projector" is one
Event with two Bookings — instead of duplicating event details per resource.

**Learn more:** [ER model (Wikipedia)](https://en.wikipedia.org/wiki/Entity%E2%80%93relationship_model) · [Lucidchart ER diagram guide](https://www.lucidchart.com/pages/er-diagrams)

## 7. Normalization (and the "no row explosion" story)

**Idea.** **Normalization** is organizing tables so each fact is stored **once**. It
avoids three problems: update anomalies (changing a fact in many places), insert anomalies
(can't add X without Y), and delete anomalies (losing data as a side effect). The first
three "normal forms" (1NF→3NF) progressively remove repetition.

**In our code.**
- A user's name lives **only** in `users`. Events reference the user by `organizer_id`,
  not by copying the name. Rename the user once → every event reflects it.
- **Recurrence is the standout example.** A class meeting Mon+Wed for 16 weeks is **not**
  ~32 rows. It's one `RecurrenceRule` row holding an RFC 5545 string
  (`FREQ=WEEKLY;BYDAY=MO,WE;UNTIL=...`) + one recurring-root `Event` + one template
  `Booking`. The individual dates are *computed at runtime*, never stored. See the long
  design comment in `RecurrenceRule` in `models.py`. This is normalization meeting
  efficiency: store the rule, not its expansion.

**Learn more:** [Database normalization (Wikipedia)](https://en.wikipedia.org/wiki/Database_normalization) · [1NF/2NF/3NF with examples](https://www.studytonight.com/dbms/database-normalization.php)

## 8. Constraints: unique, not-null, default

**Idea.** **Constraints** are rules the database itself enforces, so bad data can't get in
even if application code has a bug.

**In our code.**
- **Unique:** `email = Column(String(255), unique=True, ...)` — no two users share an email.
- **Not null:** `nullable=False` on `email`, `full_name`, `hashed_password` — these must
  always have a value. `capacity` is `nullable=True` because equipment may have no seats.
- **Default:** `is_active = Column(Boolean, default=True, ...)`, `role` defaults to
  `VIEWER`, timestamps default to `utcnow`.
- **Composite unique:** `event_participants` has a unique constraint on `(event_id, user_id)`
  so the same person can't be added to one event twice.

**Learn more:** [Postgres constraints](https://www.postgresql.org/docs/current/ddl-constraints.html)

## 9. Indexes

**Idea.** An **index** is a secondary data structure (usually a B-tree) that lets the
database find rows by a column's value *without scanning the whole table* — like a book's
index instead of reading every page. Cost: indexes use space and slightly slow writes, so
you add them only for columns you filter/join/sort on often.

**In our code.** Indexes are declared in `__table_args__`:
```python
# Resource:
__table_args__ = (Index("ix_resources_type", "resource_type"),)   # filter rooms by type
```
Other deliberate ones:
- `bookings`: composite index on `(resource_id, start_time, end_time)` — exactly the
  columns the **conflict query** filters on (section 14). This is why double-booking
  checks stay fast.
- `notifications`: `(recipient_id, is_read)` — "my unread notifications" is instant.
- `events`: indexes on `organizer_id`, `start_time`, `recurrence_rule_id`.

**Learn more:** [Use The Index, Luke!](https://use-the-index-luke.com/) (the best free index resource) · [Postgres indexes](https://www.postgresql.org/docs/current/indexes.html)

## 10. Enums in the database

**Idea.** An **enum** column accepts only values from a fixed list. It's a constraint that
documents intent and prevents "Lab" vs "lab" vs "Laboratory" drift.

**In our code.** `UserRole`, `ResourceType`, `EventStatus`, `BookingStatus`,
`NotificationType`, `FeedbackCategory` — all Python `enum.Enum` classes mapped with
`SAEnum`. The `ResourceType` docstring spells out the reasoning. Adding a new enum value is a
**schema change**: the Postgres enum type must be altered (`create_all` will **not** do this).

⚠️ **Gotcha we actually hit (Phase 2):** SQLAlchemy stores a Python enum by its member
**NAME**, not its value. So the DB labels are `CLASSROOM`, `LAB`, … (uppercase names) — *not*
`classroom`. The correct one-time SQL therefore uses the NAME:
```sql
ALTER TYPE resourcetype ADD VALUE IF NOT EXISTS 'COMPUTER_ROOM';
```
(`resourcetype` = the type name SQLAlchemy generated from `ResourceType`. Confirmed via
`SAEnum(ResourceType).enums`. Meanwhile the **API** still sends/receives the lowercase
*value* `computer_room` — Python maps name↔value for us, so the frontend is unaffected.)

**Learn more:** [Postgres enum types](https://www.postgresql.org/docs/current/datatype-enum.html)

## 11. ORM — Object-Relational Mapping (SQLAlchemy)

**Idea.** There's a mismatch between **objects** (Python classes, instances, references)
and **relations** (tables, rows, foreign keys). An **ORM** bridges them: you work with
Python objects, it generates SQL. You write `db.query(User).filter(...)`, it emits
`SELECT ... FROM users WHERE ...`.

**In our code.** SQLAlchemy 2.0. `Base` (defined in
[`database.py`](../backend/app/core/database.py)) is the registry every model inherits.
Querying looks like Python:
```python
resource = self.db.query(Resource).filter(
    Resource.id == data.resource_id,
    Resource.is_active == True,
).first()
```
**Pros:** type-safe-ish, composable, DB-portable. **Watch out for:** the *N+1 query
problem* (looping over objects and lazily loading a relationship each time fires one query
per row). Worth knowing as we add list views.

**Learn more:** [SQLAlchemy 2.0 ORM tutorial](https://docs.sqlalchemy.org/en/20/orm/quickstart.html) · [What is an ORM (Martin Fowler context)](https://martinfowler.com/bliki/OrmHate.html)

## 12. Sessions, the unit of work, and connection pooling

**Idea.** A **session** is a workspace for a set of DB operations: you load objects,
change them, and `commit()` (save) or `rollback()` (discard) as one **unit of work**.
Opening a real DB connection is expensive, so a **connection pool** keeps a set of live
connections ready to reuse.

**In our code.** [`database.py`](../backend/app/core/database.py) is small but dense:
```python
engine = create_engine(
    settings.DATABASE_URL,
    pool_pre_ping=True,   # check a connection is alive before handing it out
    pool_size=10,         # keep 10 connections ready
    max_overflow=20,      # allow 20 more under burst load (30 max)
)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

def get_db():
    db = SessionLocal()
    try:
        yield db          # hand this session to one request
    finally:
        db.close()        # always return the connection to the pool
```
**Request isolation:** every HTTP request gets its **own** fresh session via FastAPI's
`Depends(get_db)`. No two requests share mutable DB state. The `try/finally` guarantees
the connection goes back to the pool even if the request errors.

**Learn more:** [SQLAlchemy session basics](https://docs.sqlalchemy.org/en/20/orm/session_basics.html) · [Connection pooling](https://docs.sqlalchemy.org/en/20/core/pooling.html)

## 13. Transactions and ACID

**Idea.** A **transaction** groups multiple operations into one all-or-nothing unit.
**ACID** = **A**tomicity (all or nothing), **C**onsistency (rules stay satisfied),
**I**solation (concurrent transactions don't corrupt each other — section 14),
**D**urability (once committed, it survives a crash).

**In our code.** Creating an event with bookings is atomic: the event row and all its
booking rows commit together, or not at all. If a booking conflicts mid-way, the whole
thing rolls back — you never get an orphan event with no room. The session's
`commit()`/`rollback()` (with `autocommit=False`) is what makes this a real transaction.

**Learn more:** [ACID (Wikipedia)](https://en.wikipedia.org/wiki/ACID) · [Postgres transactions](https://www.postgresql.org/docs/current/tutorial-transactions.html)

## 14. Concurrency control and locking — the double-booking problem ⭐

**This is the most important DBMS concept in the app — study this one closely.**

**The problem (a race condition).** Two professors hit "book Room 601" for the same slot
at the same instant. Both requests run the "is it free?" check. Both see "free." Both
insert. Now the room is double-booked. The bug exists *only* because the two transactions
overlap in time.

**The idea — locking.**
- **Pessimistic locking:** assume conflict will happen; lock the relevant rows so the
  second transaction *waits* until the first finishes. `SELECT ... FOR UPDATE`.
- **Optimistic locking:** assume conflict is rare; let both proceed but detect the clash
  at commit (e.g., via a version column) and retry the loser.

**In our code.** We use **pessimistic locking** in `_create_booking`
([`bookings/service.py`](../backend/app/modules/bookings/service.py)):
```python
conflict = self.db.query(Booking).filter(
    Booking.resource_id == data.resource_id,
    Booking.is_recurring_template == False,
    Booking.status.in_([CONFIRMED, APPROVED, PENDING]),
    Booking.start_time < data.end_time,     # overlap test, half of it
    Booking.end_time   > data.start_time,   # the other half
).with_for_update().first()                 # 🔒 the lock
```
The two time comparisons are the classic **interval-overlap test**: two ranges overlap iff
`A.start < B.end AND A.end > B.start`. `with_for_update()` is the `FOR UPDATE` lock — it
holds the matching rows so a concurrent booking request blocks until this transaction
commits or rolls back, closing the race.

The doc comment in the code also notes the *production-grade* alternative: a PostgreSQL
`EXCLUDE` constraint over a `tsrange`, which lets the database itself forbid overlaps. We
chose row locking because it needs no raw-SQL migration.

> ✅ **Phase 0 update — read path vs write path.** This overlap logic now lives in ONE
> place: `AvailabilityService.find_conflict(resource, start, end, lock=...)` in
> [`modules/availability/service.py`](../backend/app/modules/availability/service.py).
> The `lock` flag *is* the whole lesson, made switchable:
> - **Write path** (creating a booking) calls it with `lock=True` → `SELECT ... FOR UPDATE`,
>   so two simultaneous bookings can't both win the same slot.
> - **Read paths** (room colours, empty-room search, free-slot finder) call it with
>   `lock=False` → no lock, because a read changes nothing and so can't cause the
>   double-booking race. Reads never block other users.
>
> `_create_booking` now delegates here too, so "can I book?" and "is it free?" can never
> disagree — one rule, two callers.

**Isolation levels** (brief): databases offer levels (Read Committed → Serializable)
trading strictness for concurrency. Postgres defaults to **Read Committed**; our row lock
adds the extra safety we need for booking without going full Serializable.

**Learn more:** [Postgres explicit locking (`FOR UPDATE`)](https://www.postgresql.org/docs/current/explicit-locking.html#LOCKING-ROWS) · [Postgres transaction isolation](https://www.postgresql.org/docs/current/transaction-iso.html) · [Race conditions in DBs (martin kleppmann, DDIA ch.7)](https://www.oreilly.com/library/view/designing-data-intensive-applications/9781491903063/)

## 15. Soft deletes and append-only tables

**Idea.** Sometimes you must *not* truly delete data — for audit, history, or because other
rows still reference it. **Soft delete** = a flag (`is_active=False`) instead of `DELETE`.
**Append-only** = rows are only ever inserted, never updated or deleted.

**In our code.**
- **Soft delete:** "Delete resource" actually sets `is_active=False`. Past bookings that
  reference it stay valid; the room just stops appearing in pickers.
- **Append-only:** `audit_logs` records every action and is never updated — the design
  comment says so explicitly. `feedback` is similar.

**Learn more:** [Soft delete patterns](https://www.cybertec-postgresql.com/en/soft-delete-in-postgresql/) · [Audit logging](https://en.wikipedia.org/wiki/Audit_trail)

## 16. Schema migrations (Alembic)

**Idea.** The schema changes over a project's life (new column, new table, new enum value).
A **migration** is a versioned, repeatable script that moves the database from one schema
version to the next — and back down if needed. It's "version control for your schema."

**In our code.** Alembic is configured under `backend/alembic/`. The workflow (from
`CLAUDE.md`):
```bash
alembic revision --autogenerate -m "add groups and roster tables"  # diff models → script
alembic upgrade head     # apply
alembic downgrade -1     # roll back one step
```
**Reality check for *this* project (found in Phase 2):** there are currently **no Alembic
version files** — the schema is built by `Base.metadata.create_all()` in `app/main.py`,
which runs on every startup. Consequences:
- **New tables** (groups/roster, release requests) are **auto-created on the next backend
  restart** — `create_all` adds missing tables. No migration needed.
- **Changes to existing tables** (a new enum value like `computer_room`, or a new column)
  are **NOT** applied by `create_all` — it never alters existing tables. Those need a
  one-line `ALTER` (see §10) run once, or recreating the dev DB.
- For production you'd switch to real Alembic migrations (the commands above).

**Learn more:** [Alembic tutorial](https://alembic.sqlalchemy.org/en/latest/tutorial.html) · [Why migrations matter](https://martinfowler.com/articles/evodb.html)

## Quick glossary

| Term | One-liner |
|---|---|
| Relation / table | A typed set of rows |
| Tuple / row / record | One entry in a table |
| Attribute / column / field | One typed property of a row |
| PK | Column that uniquely identifies a row |
| FK | Column pointing at another table's PK |
| Junction table | Extra table implementing a many-to-many link |
| Index | Lookup structure to avoid full-table scans |
| Transaction | All-or-nothing group of operations |
| ACID | Atomicity, Consistency, Isolation, Durability |
| Pessimistic lock | Lock rows up front so others wait (`FOR UPDATE`) |
| ORM | Maps objects ⇄ relational rows (SQLAlchemy) |
| Migration | Versioned schema-change script (Alembic) |
| Normalization | Storing each fact exactly once |
